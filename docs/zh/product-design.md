# HyClaw - 产品设计文档

## 一、产品定位

多 Agent 协作的编程工作台。不同于传统 AI 编程工具的一对一私聊模式，HyClaw 以项目管理 Agent，以群聊模式驱动多 Agent 协作，每个 Agent 独立记忆、单线程工作、产出统一由主 Agent 合并。

## 二、核心概念

### 三层架构

```
Project "项目名"
├── Agent 池 (角色定义，属于项目)
│   ├── PM       (系统自带，默认成员)
│   ├── 自定义Agent A
│   ├── 自定义Agent B
│   └── ...
│
├── 对话 1
│   ├── 👤 我
│   ├── PM (实例)
│   ├── Agent A (实例)
│   └── Agent B (实例)
│
├── 对话 2
│   ├── 👤 我
│   ├── PM (实例)
│   └── Agent A (实例)  ← 同一角色，不同对话，独立实例
│
└── 对话 3 ...
```

- **Agent = 角色配置**（类似 class）：定义技能、性格、工作流，属于项目级别
- **Agent in 对话 = 实例**（instance）：角色在某个对话中的具体化身，独立记忆、独立任务队列
- **同一 Agent 可同时存在于多个对话**，各自互不干扰

### PM Agent (主 Agent)

- 系统自带，每建一个项目自动生成
- 职责：对接用户需求 → 分解任务 → 分发给各 Agent → 审核产出 → 合并代码 → 处理冲突
- 所有文件操作最终由 PM 统一写入，防止多 Agent 争抢文件

## 三、群聊交互模型

### 连续对话

用户可以连续发多条消息，PM 逐条引用回复：

```
你 10:00 → "@PM 帮我看看项目结构"
你 10:01 → "顺便把 README 也更新一下"
你 10:01 → "日志改成用 pino"

PM 10:00 → [回复"帮我看看项目结构"] 项目结构如下：src/下有...
PM 10:01 → [回复"顺便把 README 也更新一下"] ✅ 已更新 README
PM 10:01 → [回复"日志改成用 pino"] ✅ 已替换 winston → pino，改了 12 个文件
PM 10:02 → [汇总卡片] 3 个任务完成，改动预览...
```

### 任务分发与并行

用户一条消息可以触发 PM 同时分发给多个 Agent 并行工作：

```
你：@PM 写登录页面 + 写爬虫抓首页

PM → @前端家 你负责登录页面
PM → @爬虫家 你负责爬虫
前端家 → [写代码中...]
爬虫家 → [抓取中...] 遇到问题追问用户
PM → 协调、合并、汇报
```

### 引用机制

所有回复都带引用，形成可追踪的任务链：
- 用户消息 → PM 回复（引用）
- Agent 追问 → PM 回复（引用）
- 任务完成 → 汇总卡片

## 四、Agent 管理

### 创建 Agent

用户用自然语言描述，由模型生成 Agent 配置：

```
你：我要一个资深的 Python 后端工程师，精通 FastAPI 和 SQLAlchemy，
    代码风格偏好 type hints 和 pydantic，注释用中文，变量用英文。
    工作方式：先出方案 → 确认 → 写代码 → 自测。性格沉稳不啰嗦。

PM：✅ Agent "后端老王" 已创建，加入 Agent 池
```

### Agent 属性（由提示词自然语言定义）

| 属性 | 说明 |
|------|------|
| 角色 | 它是谁，擅长什么 |
| 技能栈 | 具体技术栈 |
| 工作流 | 怎么做事的 |
| 性格 | 说话风格 |
| 权力边界 | 能做什么不能做什么 |
| 记忆偏好 | 记什么不记什么 |

### Agent 操作

- **克隆**：复制角色配置，修改技能
- **调教**：用自然语言对话调整行为
- **暂停/恢复**：临时停用某 Agent
- **删除**：从项目移除

## 五、任务系统

### Agent 单线程模型

每个 Agent 实例一次只专注做一个任务，新任务排队。

```
后端老李 的任务面板：
┌─────────────────────────────────┐
│ 🔄 进行中                        │
│  #7 实现用户认证接口    [暂停]    │
│                                  │
│ ⏳ 队列中                        │
│  #8 重构数据库模型      [取消]    │
│  #9 添加缓存层                   │
│                                  │
│ ⏸ 阻塞中                        │
│  #10 实现缓存层 ← 等 #7          │
│                                  │
│ ✅ 已完成                        │
│  #6 修复登录 Bug                 │
└─────────────────────────────────┘
```

### 优先级（三种插队模式）

| 级别 | 行为 |
|------|------|
| 🔵 普通 | 排到队尾 |
| 🟡 优先 | 插到进行中任务后面 |
| 🔴 立即 | 暂停当前，做完恢复 |

### 任务状态

```
已排队 → 进行中 → 待审核 → 已完成
           │         │
           ├─ 暂停    ├─ 驳回 → 回到排队
           ├─ 等待中（等用户回复）
           └─ 阻塞（等另一个 Agent）
```

### 跨 Agent 依赖

```
PM：@后端老李 #7 认证接口
PM：@前端老王 #8 登录页面 ← 依赖 #7

前端老王：#8 检测到依赖 #7，等待中...
         后端老李完成后自动开始

── #7 完成后 ──
前端老王：依赖满足，自动开始 #8
```

## 六、文件工作区隔离

### 核心模型：对话 = Git 分支

```
Project "电商平台" (Git Repo)
├── main                     ← 唯一真相
├── conv/login               ← 对话"登录模块"的独立分支
└── conv/crawler             ← 对话"首页抓取"的独立分支

合并流：
  Agent 产出 → PM 审核 → 对话分支 → PM 合并 → main
```

### 冲突预防

- 所有 Agent 不直接写主工作区
- PM 顺序合并，冲突时生成 diff 对照，交给用户裁决
- 底层用 git 追踪

### 三个合并策略

| 情况 | PM 行为 | 用户体验 |
|------|---------|---------|
| 无冲突 | 自动合，不询问 | 通知"已合并" |
| 简单冲突（同文件不同区域） | 自动合，告知合并内容 | 通知 + 一键回退 |
| 复杂冲突（同文件同区域） | 暂停，@用户裁决 | 并排对比，选 A/B/手动 |

### 对话状态指示

每个对话窗口顶部显示工作区状态：
- 📁 干净 ✅ 已同步
- 📁 N个改动 ○ 无冲突
- 📁 N个改动 ⚠ 有冲突

## 七、内置功能模块

### 浏览器

| 模式 | 说明 |
|------|------|
| 无头（Agent 用） | Agent 后台抓取，发截图和数据到群里 |
| 有头（用户用） | 右侧面板弹出浏览器，用户和 Agent 共享视图 |

### 文档编辑器

- 产出为可交互的"文档卡片"，而非文件路径
- Block-based 编辑器（类似 Notion/TraceSolo）
- 支持：代码块、表格、图片、流程图、公式
- 用户和 Agent 可协作编辑，Agent 改动须用户确认

### 视频编辑

- 核心能力：拼接、裁剪、转码、加字幕/水印、调整分辨率/帧率
- 产出预览卡片，可播放后确认导出

### 自动化 & 定时任务

```
定时：每天早上 9:00 爬虫家 → 抓取竞品 → 文档家 → 汇总报告 → 发到群里
事件：git tag push → 生成 CHANGELOG → 推送到飞书群
```

## 八、产品交互黄金规则

| 规则 | 说明 |
|------|------|
| 群聊即协作 | 对话是多 Agent 协作空间，不是一对一提问 |
| PM 是管家 | 用户只跟 PM 说，PM 负责分配和协调 |
| Agent 是独立人格 | 可创建、调教、暂停、克隆、删除 |
| 所有产出都是消息 | 代码、文档、视频都是群聊消息卡片 |
| 引用即追踪 | 每条消息都可追溯到原始需求 |
| 冲突交给 PM | 用户看不到 git 细节，看到的是 PM 的处理消息 |

## 九、界面布局

### 三栏布局

```
┌──────────┬────────────────────┬──┬──────────────┐
│ Sidebar  │    Main Content     │  │  ToolPanel   │
│          │                    │  │              │
│ 项目列表  │  对话 / 设置页       │  │  浏览器      │
│ 对话列表  │                    │  │  任务        │
│ Agent池  │                    │  │  Agent       │
│          │                    │  │  文档        │
│ ⚙️ 🌓   │                    │  │  视频        │
│          │                    │  │  自动化      │
└──────────┴────────────────────┴──┴──────────────┘
        可拖拽调整            固定贴右  展开/收起
```

- **左侧栏**：项目对话导航，可折叠
- **中间主区域**：对话 / 项目设置页切换
- **右侧工具栏**：图标列固定贴右边缘，点击展开面板，再次点击 / 点 X 关闭

### 空状态设计

| 场景 | 显示 |
|------|------|
| 无项目 | 欢迎页：Logo + 三卡片介绍 + "新建项目"按钮 |
| 有项目无对话 | 右侧对话区空状态提示 + 引导创建对话 |
| 有项目无 Agent | 项目设置页提示 + 引导创建 Agent |

### 项目设置页

点击侧栏顶部项目名 → 主区域切换为设置页：
- 项目信息：名称编辑、工作目录
- Agent 成员列表：完整增删改查，PM 为系统 Agent 不可删除
- 左侧 ← 返回对话

## 十、模型设置

### 设计原则
- 接口统一为 **Anthropic API 兼容格式**，用户不感知底层协议差异
- 只提供：选择服务商 → 选择模型 → 填写 Key
- API 地址由系统根据服务商自动填充，用户不可见

### 服务商预设

| 服务商 | API 地址（系统预设） | 默认模型 |
|--------|---------------------|---------|
| Anthropic | api.anthropic.com/v1 | claude-sonnet-4-20250514 |
| Kimi（月之暗面） | api.moonshot.cn/anthropic | kimi-k2.6 |
| GLM（智谱） | open.bigmodel.cn/api/paas/v4 | GLM-5.1 |
| DeepSeek | api.deepseek.com/v1 | deepseek-v4-pro |
| MiniMax | api.minimaxi.com/v1 | MiniMax-M2.7 |

### 交互
- 支持添加多个服务商（列表展示）
- ○ 单选当前使用的服务商
- 🔑 编辑仅允许修改 API Key
- 🗑 删除服务商，自动切换到下一个

## 十一、数据持久化

### 技术
- 使用 **Dexie.js**（IndexedDB 封装）
- 数据库名：`hyclaw`，版本 2
- 五张表：projects / agents / conversations / messages / tasks
- 一张 meta 表：存储活跃项目/对话 ID

### 启动恢复
1. 应用启动 → 显示欢迎页（无数据闪一下）
2. `hydrateFromDB()` 从 IndexedDB 加载所有数据
3. 恢复上次活跃的项目和对话
4. 所有 mutation 操作同步写入 IndexedDB

### 主进程数据
- 模型设置：`~/hyclaw-data/settings.json`
- 定时任务：`~/hyclaw-data/scheduler-tasks.json`
- 视频输出：`~/hyclaw-output/`
